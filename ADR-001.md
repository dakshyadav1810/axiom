# ADR-001: Restrict LLM Usage to Authoring & Maintenance

**Status:** Accepted  
**Date:** 2026-07-07  
**Author:** Shubham Krishan

## Context

Axiom is an intent-driven testing framework that uses LLMs to translate high-level user intent into executable tests.

During the initial architecture exploration, runtime LLM involvement was considered for self-healing. The proposed flow allowed failed tests to generate a repair artifact, invoke an LLM, regenerate the test specification, and retry automatically.

While technically feasible, this introduces significant drawbacks:

- Runtime becomes nondeterministic.
- Every failed test incurs additional latency and token cost.
- Large test suites become expensive to execute.
- Runtime behavior becomes difficult to reason about and debug.
- The framework becomes dependent on LLM availability during execution.

The primary goal of Axiom is reliable, scalable, deterministic test execution. LLMs should therefore be treated as an **authoring tool**, not a runtime dependency.

## Decision

LLMs will only participate during **authoring and explicit maintenance**.

They are responsible for:

- Translating natural language test intent into a semantic test specification.
- Generating UI, API, and database assertions.
- Updating stale tests during explicit repair initiated by the developer.

Runtime execution must remain fully deterministic.

The runtime pipeline is:

```text
Cached Selector
        │
        ▼
Deterministic Resolvers
        │
        ▼
Testing Adapter
        │
        ▼
Browser
```

If execution fails:

1. Attempt cached selector.
2. Attempt deterministic resolver-based healing.
3. If deterministic healing fails, mark the test as **stale** and surface it for manual repair.

An LLM is **not** invoked automatically during runtime.

## Options Considered

### Runtime LLM Healing

**Pros**

- Can reason about arbitrary UI changes.
- Potentially higher recovery rate.

**Cons**

- Nondeterministic execution.
- High runtime token cost.
- Increased latency.
- Difficult to scale.
- Difficult to audit and reproduce.

**Decision:** Rejected.

---

### Deterministic Runtime + LLM Authoring (Chosen)

**Pros**

- Deterministic execution.
- Predictable runtime cost.
- Fast repeated executions.
- Easily reproducible failures.
- LLM costs shifted to authoring and maintenance where human review is expected.

**Decision:** Accepted.

## Consequences

- Runtime execution remains deterministic.
- Resolvers become the primary self-healing mechanism.
- Cached selectors remain the first lookup strategy.
- LLM-generated repairs become an explicit maintenance workflow rather than an automatic runtime feature.
- Future optimizations (repository indexing, semantic knowledge graphs, compact IR formats, etc.) remain compatible with this architecture, but do not affect the runtime execution model.

## Out of Scope

The following topics are intentionally left outside this ADR and will be documented separately:

- Intent IR format
- Repository indexing / Knowledge Graph
- Resolver architecture
- Semantic execution plan
- Repair artifacts
- Token optimization strategies

These are implementation details rather than architectural decisions.

## Design Principle

> Runtime should be deterministic. Intelligence belongs in authoring, not execution.

This principle serves as the foundation for future architectural decisions within Axiom.
