# ADR-003: Single-Language TypeScript Stack — Node Core, Bundled Dashboard, MCP-in-CLI

**Status:** Accepted
**Date:** 2026-07-10
**Author:** Shubham Krishan
**Relationship:** Fixes the implementation stack for ADR-001 (deterministic runtime) and ADR-002 (spec/grounding/MCP pipeline). Detailed in [PLAN-001.md](../PLAN-001.md). Supersedes the language/runtime choices previously recorded in AGENTS.md.

## Context

AGENTS.md and DECISIONS.md describe a three-language platform: a Bun/TypeScript CLI, a ~21.6k-line Python core (FastAPI, Playwright, Pydantic, aiosqlite, Supabase, rapidfuzz), and a Next.js dashboard shipped as a separate deployable package.

That split imposes a permanent tax that this project, specifically, does not need to pay:

- The usual reason to keep Python is scientific/ML coupling. Axiom has almost none — across 21.6k lines the only pythonic dependency is `rapidfuzz`, in a single resolver file. Per ADR-001, all generative-LLM and semantic-authoring work happens at authoring time via API calls, not local Python models.
- Playwright is natively a Node library; the Python package drives the Node Playwright over a subprocess pipe. A testing tool built *on* Playwright is better off in Playwright's native language.
- The recorder/resolver already lives partly in JavaScript — there are 34 `page.evaluate` sites where logic is shipped into the page as strings. In TypeScript those become typed, testable, bundled modules.
- The system is IR/artifact-driven (`spec.json`, `candidates.json`, KDG, config). A three-language stack forces the same schema to be maintained as Pydantic in core *and* Zod in the dashboard, kept in parity by hand.

The project is pre-1.0 (CLI is a stub, dashboard scaffolded, ADRs still in flight, band thresholds untuned), so the cost of consolidating now is far lower than living with the tax indefinitely.

## Decision

Adopt a single-language **TypeScript-on-Node** stack, built from scratch. The existing Python core is treated as an executable reference specification, not code to be preserved.

1. **One language, one runtime.** Node.js LTS (22+) + TypeScript across all packages. `npx axiom` is the only install step — no Python, venv, or `uv`.
2. **Shared IR contract.** A new `packages/shared` holds the Zod schemas (Spec IR, candidates, resolver models, config, REST DTOs, WS messages) — defined once and imported by core, cli, and dashboard. Zod replaces Pydantic.
3. **Core owns execution, in TypeScript.** `packages/core` is a **Fastify** server owning all business logic: Playwright (native), the Multi-Signal Resolver, recording, grounding, execution, storage, REST + WebSocket. SQLite via **better-sqlite3 + Drizzle** for the ephemeral cache.
4. **Semantic search replaces fuzzy matching.** The semantic resolver signal uses a **local, fixed-weight ONNX embedding model** (e.g. transformers.js / fastembed) plus in-process cosine similarity, with embeddings cached in SQLite. `rapidfuzz`/fuzzy-string matching is removed entirely — it was a workaround for the absence of semantic search. The model is deterministic and makes no generative/network call, so this remains consistent with ADR-001's LLM-free runtime.
5. **The dashboard is bundled inside the core server.** `packages/dashboard` is a bare-minimum **Vite + React SPA** built to static assets that Fastify serves directly. It is not a separately deployable package. Deliberately minimal — no terminal emulator, no animation library.
6. **The CLI is the Node orchestration + MCP control plane.** `packages/cli` (commander) spawns and supervises the core server and hosts the agent-facing MCP server (official TypeScript SDK) started by `axiom start`. MCP tools translate to REST/WS calls into core — never internal cross-package calls (invariant #7).

## Options Considered

### A. All-TypeScript (chosen) vs. keep Python core vs. incremental strangler
- **Keep Python core (CLI/MCP only in TS):** zero rewrite risk, but a permanent two-language tax on a schema-heavy project whose Python buys almost nothing here.
- **Incremental strangler:** port domain-by-domain behind the REST contract. Lower short-term risk, but doubles the surface (two cores running) for months on a pre-1.0 codebase.
- **Chosen — from-scratch all-TypeScript:** one language and runtime, one shared IR, single-command install. Cost is porting ~21.6k lines of already-tuned logic; acceptable because the project is pre-1.0 and the Python behavior serves as a reference to test against.

### B. Semantic search via local embeddings (chosen) vs. fuzzy string matching
- **Fuzzy (`rapidfuzz`/`fuzzball`):** cheap, but a lexical workaround that misses semantic equivalence — the exact failure the resolver's "semantic" signal is meant to cover.
- **Chosen — local embedding model + cosine:** true semantic similarity, deterministic (fixed offline weights, no API), embeddings cached in SQLite. Consistent with ADR-001.

### C. Dashboard bundled in core (chosen) vs. separate Next.js deployable
- **Separate Next.js package:** independent deploy and SSR, neither of which a local, client-only developer tool needs.
- **Chosen — Vite SPA served by Fastify:** one process, one install, no SSR runtime; matches "the dashboard is only a client."

### D. MCP server in the CLI (chosen) vs. in the core
- **MCP in Python/TS core (FastMCP-style, in-process):** no network hop, but makes the CLI a thin shim and puts a second interface directly inside core.
- **Chosen — MCP in the Node CLI:** the CLI is the single process a developer/agent talks to, orchestrating lifecycle and proxying MCP tool calls to core over REST/WS, preserving invariant #7.

## Consequences

- The Python core, `pyproject.toml`, `uv.lock`, and venv setup are retired; the `packages/core/**/*.py` tree remains only as a reference during the rewrite and is deleted once parity is verified.
- `packages/shared` becomes the single source of truth for the IR; drift between core and dashboard schemas is eliminated by construction.
- Onboarding collapses to installing Node and running `npx axiom` — no Python toolchain.
- Resolver behavior (the tuned ladder in DISCUSSION.md) must be re-earned in TypeScript and locked in with golden-case Vitest tests during the port.
- The semantic signal now carries a small local model as a runtime dependency (bundled weights + ONNX runtime); startup and memory budgets in AGENTS.md must account for it, and embeddings are cached to keep repeated runs fast.
- AGENTS.md's package table and the Bun-oriented CLI `CLAUDE.md` are now stale and are updated to match this ADR.

## Design Principle

> One language, one runtime, one IR. Intelligence stays in authoring; the deterministic core, its dashboard, and its MCP control plane are all TypeScript.
