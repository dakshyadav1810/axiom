# ADR-002: Test Authoring Pipeline — Spec IR, Grounding, and the Healing Loop

**Status:** Accepted  
**Date:** 2026-07-09  
**Author:** Shubham Krishan  
**Relationship:** Operationalizes ADR-001 — fills in the "Intent IR format", "Semantic execution plan", and "Repair artifacts" it left out of scope.

## Context

ADR-001 established the principle — LLMs author, runtime is deterministic — but deliberately left the concrete pipeline undefined. This ADR defines how a natural-language intent becomes a deterministic, self-healing, version-controlled test.

The governing constraint (from analysis of the previous `axiom-resolvers` system): the resolver matches an authored **target** against **candidates** extracted from the live DOM. These are two different sources of truth. An LLM can author the *semantic* half of a target (label, role, meaning) with no page access, but it cannot reliably fabricate the *structural* half — CSS/XPath selectors, geometry, DOM structure — which only exists once a real page is rendered. Any pipeline that asks the LLM to emit selectors is non-deterministic at exactly the point that must be reproducible.

We therefore need an explicit step that grounds LLM-authored intent against a real DOM, and a clear separation between the durable intent (versioned, rarely changes) and the volatile DOM anchors (which drift with every UI change).

## Decision

Compile intent into a test through a staged pipeline that keeps durable intent and volatile grounding in **separate artifacts**, combined deterministically before resolution. Each stage is independently replaceable.

```text
Prompt {intent, entry, vars, assertions?}
        │  (LLM, authoring — fetches KDG via MCP)
        ▼
spec.json          ← Spec IR: intent + semantics + assertions   (Tier-1, DOM-blind)
        │  (Grounding / first live run — deterministic)
        ▼
candidates.json    ← DOM candidates + selectors/identifiers      (Tier-2, from live DOM)
        │  (Normalize: spec.json + candidates.json → resolver input)
        ▼
Resolver → Resolution Result (winner + band) → Cache (SQLite)
        │  (Execution — deterministic, LLM-free)
        ▼
Playwright Adapter → Browser → Test Result → Dashboard (WebSocket stream)
```

### 1. Spec IR (`spec.json`) — LLM-authored, DOM-blind

The LLM translates the prompt into an ordered list of steps. Each step is an **action node** whose target carries only Tier-1 (semantic) fields:

- Step: `id`, `action` (`navigate | click | type | select | keypress | submit | wait`), `value`, `intent`, `generalization` (`same_element | any_matching | aggressive | flexible`), `preconditions`, `expectedOutcome`, `assertions[]`, `onFailure` (`abort | continue | retry_once | optional`).
- Target (Tier-1): `label`, `semantics[]`, `role`, `actions[]`, `intent`.

`spec.json` is the durable, human-reviewable, version-controlled source of truth. It contains no selectors. Assertions cover UI, API, and DB checks (DB testing is in scope for the unified model).

### 2. Grounding / First Run (`candidates.json`) — deterministic DOM capture

A single deterministic live run launches the browser at the entry URL and, for each step, extracts the interactive DOM, runs the resolver against the Tier-1 target, and records the ranked candidates plus their selectors/identifiers, geometry, and structure. The winner is accepted only if its confidence **band ≥ medium**; otherwise the step is marked `ungrounded` and surfaced for author review rather than guessing. Grounding writes Tier-2 anchors and a `cachedSelector` — it never invents them.

### 3. Normalize + Resolve + Cache

`Normalize` merges `spec.json` and `candidates.json` into the resolver's input shape. The resolver produces a Resolution Result (per-candidate signal scores, a winner, and a band). The chosen `cachedSelector` is cached in **SQLite** — the fast path for subsequent runs. Versioned artifacts (`spec.json`, the grounded test) live in JSON; SQLite holds only ephemeral, regenerable cache state.

### 4. Execution — deterministic, LLM-free

A single Playwright adapter drives the browser from the grounded test. Per ADR-001, no LLM participates. The dashboard tracks progress over a **WebSocket stream** (run/step/resolver/log events). Runtime resolution order is: **selector cache → grounded-artifact selector → deterministic resolver heal → stale** (the cache, keyed by the page's `domHash`, is the fast path; the versioned artifact's selector is the durable fallback when the cache is cold or the page changed).

### 5. The two-level healing loop

- **Runtime heal (algorithmic, LLM-free):** when a test run (`npx axiom test`) fails to locate an element, an algorithmic loop re-runs from the Grounding step. If it recovers, the change is flagged and logged. If it cannot, the step is marked for **author review** and handed to the maintenance flow. This satisfies ADR-001's "resolver-based healing, then stale."
- **Maintenance heal (explicit, LLM-assisted):** a developer-triggered flow builds a **Repair Payload = { Spec IR, Test Case, KDG }**, sends it to the LLM, and re-runs Tier-1 authoring → Grounding → Normalize to produce a corrected test. This is the only place an LLM re-enters after authoring.

### 6. MCP server as the control plane; KDG as authoring context

`axiom start` launches an MCP server that is the agent-facing interface. It exposes `getMap` (knowledge map), `getDelta` (map diff), and `healing` (returns repair context: Spec IR + Test Case + KDG), plus CRUD tools to run, report, poll, update (heal), and delete tests. At authoring time the LLM fetches the **versioned Knowledge Graph (KDG)** via an MCP tool call to ground its semantics in the app's real structure. The KDG records two kinds of codebase information — **conditionals** (including conditional features of an element) and **parent/child** relationships.

**Clarification: "the LLM" above is always the developer's own connected coding agent (Claude Code, Cursor, ...), never a model Axiom calls itself.** Axiom holds no provider client and no API key anywhere in its config. The MCP control plane exists precisely so the agent can read `getMap`/`healing` context and submit an already-authored spec via `submitSpec`/`updateTest` — there is no second, core-side authoring path for a bare CLI invocation to fall back to. See LLD-002 and LLD-006 §6.

## Artifact & data model

- **Confidence bands:** `score ≥ 0.5` → medium, `score ≥ 0.7` → high; below 0.5 → low. Grounding accepts `≥ medium`; runtime accept/stale uses the same banding.
- **Relationship:** one Spec IR ⇒ one Test Case. A collection of Spec IRs ⇒ intents ⇒ test cases ⇒ a test suite.
- **Persistence:** intent + grounded tests are versioned JSON; SQLite caches resolution/selectors that need not be versioned.

## Options Considered

### A. Split `spec.json` + `candidates.json` (chosen) vs. single `flow.json` blob vs. LLM-authored selectors
- **Single `flow.json`:** one artifact, but mixes durable intent with volatile DOM anchors, so every UI change churns the whole file and repair can't target one layer.
- **LLM-authored selectors:** no grounding step, but puts non-deterministic, hallucination-prone selectors on the critical path — the exact thing ADR-001 forbids at runtime.
- **Chosen:** the LLM authors intent only; a deterministic grounding pass attaches selectors from the real DOM; the two artifacts version and repair independently.

### B. Runtime heal restarts from Grounding (chosen) vs. cached-selector-only vs. from-scratch/LLM
- **Cached-selector-only:** fastest, but no recovery when the selector breaks — every drift becomes a failure.
- **From-scratch / LLM at runtime:** highest recovery, but non-deterministic and rejected by ADR-001.
- **Chosen:** re-run the deterministic Grounding + resolver loop; recover-and-log or escalate to author review — deterministic, and reuses the authoring code path.

### C. SQLite cache (chosen) vs. JSON files vs. fully versioned DB
- **JSON files:** simple, but conflate regenerable cache with versioned artifacts and are awkward for concurrent read/update.
- **Fully versioned DB:** heavy, and versions data that is trivially regenerable.
- **Chosen:** SQLite for ephemeral cache (selectors, resolution results); JSON-in-git for the durable spec and grounded test.

### D. MCP server as the control plane (chosen) vs. direct CLI vs. REST-only
- **Direct CLI / REST-only:** works for humans and the dashboard, but the primary consumer is a coding agent that needs tool-shaped access to the map, deltas, and healing context.
- **Chosen:** an MCP server (started by `axiom start`) as the agent interface, alongside the REST/WebSocket surface the dashboard already uses.

## Consequences

- Grounding adds a mandatory first live run before a test is executable; a spec that never grounds is not runnable.
- `ungrounded` steps require human review — the pipeline refuses to guess a selector, trading some autonomy for zero silent-wrong grounding.
- Repair is layer-targeted: a broken selector re-grounds without re-authoring intent; a changed intent re-authors without discarding grounding history.
- The band thresholds (0.5 / 0.7) are an initial heuristic, untuned against real corpora; the optimisation goal is to **reduce false positives** (a test that passes/locates the wrong element), and thresholds will move as data arrives.
- The KDG's internal data structure is **not yet decided** (candidate: a 3D/typed tree capturing conditionals + parent/child); this ADR fixes what the KDG must contain and how it is consumed, not how it is stored.
- Dashboard visual design, full MCP tool schemas, and coverage metrics are named here but specified elsewhere.
- Reaffirms ADR-001: no runtime LLM. It refines AGENTS.md's older "AI verification when confidence is low" wording — low confidence resolves to *stale / author-review*, not an automatic LLM call.

## Design Principle

> Durable intent and volatile grounding are separate artifacts. The LLM authors intent, the browser grounds it, and only deterministic code runs at test time.
